# Plan: Serve Static Files in Gunicorn Deployment via WhiteNoise

## Context

Gunicorn does not serve static files on its own. The project has a minimal SPA frontend (`static/app.js`, `static/style.css`, `templates/index.html`) that must be reachable in non-local deployments. Additionally, the Dockerfile never copies `static/` or `templates/` into the image — they only exist because local dev uses a full volume mount (`./:/code`), which won't be present on the deployment platform.

The fix is WhiteNoise: it plugs into Django's middleware stack and serves compressed, cache-busted static files directly from the gunicorn process with no extra infrastructure.

---

## Steps

### Step 1 — `requirements/base.txt`
Add whitenoise under the Deployment section:
```
# Deployment
gunicorn==25.3.0
whitenoise==6.9.0
```

### Step 2 — `incubyte_salary_management_backend_service/settings.py` — Middleware
Insert `WhiteNoiseMiddleware` immediately after `SecurityMiddleware` (WhiteNoise hard requirement):
```python
MIDDLEWARE = [
    "corsheaders.middleware.CorsMiddleware",
    "django.middleware.security.SecurityMiddleware",
    "whitenoise.middleware.WhiteNoiseMiddleware",    # add here
    "django.contrib.sessions.middleware.SessionMiddleware",
    ...
]
```

### Step 3 — `incubyte_salary_management_backend_service/settings.py` — STORAGES
Switch staticfiles backend to WhiteNoise's hashing+compression storage for non-local environments. Local keeps the default backend — `CompressedManifestStaticFilesStorage` requires a `staticfiles.json` manifest (generated by `collectstatic`) which is never run locally, and would crash `{% static %}` template tags:
```python
if ENVIRONMENT == LOCAL:
    _staticfiles_backend = "django.contrib.staticfiles.storage.StaticFilesStorage"
else:
    _staticfiles_backend = "whitenoise.storage.CompressedManifestStaticFilesStorage"

STORAGES = {
    "default": {
        "BACKEND": "django.core.files.storage.FileSystemStorage",
    },
    "staticfiles": {
        "BACKEND": _staticfiles_backend,
    },
}
```
In non-local environments, `collectstatic` produces content-hashed filenames (e.g. `app.abc123.js`) with pre-compressed `.gz`/`.br` variants. `{% static 'app.js' %}` resolves through the manifest automatically.

### Step 4 — `Dockerfile`
Copy `static/` and `templates/` into the image (before the `chown` so permissions are fixed):
```dockerfile
# Copy only required Django code
COPY manage.py .
COPY incubyte_salary_management_backend_service/ ./incubyte_salary_management_backend_service/
COPY apps/ ./apps/
COPY server.sh .
COPY static/ ./static/
COPY templates/ ./templates/

# Fix permissions
RUN chown -R appuser:appgroup /code
```

### Step 5 — `server.sh`
Run `collectstatic` before gunicorn starts (not for local — runserver handles static files natively):
```bash
else
    echo "Collecting static files..."
    python manage.py collectstatic --noinput

    echo "Running gunicorn server on ${BIND_ADDRESS}"
    exec gunicorn ...
```
`--noinput` prevents the interactive prompt from hanging container startup. `set -e` already ensures a failed collectstatic aborts before gunicorn starts.

---

## How It Works End-to-End

1. Docker build copies `static/` and `templates/` into the image
2. Container starts → `collectstatic` runs → hashed files written to `staticfiles/`
3. Gunicorn starts → WhiteNoise middleware intercepts all `GET /static/*` requests, serves pre-compressed files with immutable cache headers — Django view code never runs for static files
4. Template renders → `{% static 'app.js' %}` resolves to hashed URL via manifest

Local development is unchanged: `runserver` serves files directly from `STATICFILES_DIRS`, `collectstatic` never runs.
